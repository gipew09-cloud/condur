## Кто мы и что делаем

Я — владелец небольшой компании грузоперевозок (рефрижераторы, СПб): возим
продукты со склада «Агропарк Софийская 151» по РЦ сетей (Лента, Магнит,
Перекрёсток…). Я НЕ программист — объясняй просто, работай блоками, не
марафоном. Мы строим **condur** — собственную систему управления автопарком:
«умный диспетчер + финансовый контролёр». Цель — абсолютный контроль: каждая
машина, каждый километр, каждый рубль; система сама замечает подозрительное.

**Из чего состоит:** два Telegram-бота (владельца и водителя) + веб-кабинет +
GPS-интеграция Stavtrack (трекеры в машинах, ретрансляция ЕГТС на наш сервер).

**Стек (ЗАМОРОЖЕН, не менять):** Python 3.11, aiogram 3, FastAPI + Jinja2 +
HTMX + Pico.css, PostgreSQL (SQLAlchemy 2 async), Alembic (head = 0012),
APScheduler, openpyxl, Яндекс.Карты (JS API, ключ в env YANDEX_MAPS_API_KEY).
Код: локально `~/Downloads/tms-mvp`, GitHub `gipew09-cloud/condur`, деплой
Railway (2 сервиса из одного репо: `condur` = боты+сайт, `egts-receiver` =
GPS-приёмник, роль через SERVICE_ROLE=egts). Git коммитит владелец сам.
Бюджета на платные API нет. Биллинг заморожен (BILLING_ENABLED=off).

**Документы в репо (прочитай перед работой):** `HANDOFF.md` — журнал всего
сделанного; `DEV_TEST_GUIDE.md` — устройство проекта; `STAVTRACK_INTEGRATION.md`
— GPS; `GPS_IDEAS_PROMPT.md` — идеи. Тесты: `pytest` (62 passed; локальный
venv-tools на Python 3.9 — поэтому веб-router в тестах НЕ импортируется,
логику выносим в чистые сервисы: timeutil, rc_service, egts, telemetry_service).

## Что уже работает (кратко)

Боты: смены/рейсы/расходы/фото одометра, выручка с подтверждением владельцем
(pending), скрытая команда /wipe (полный сброс всего, включая аккаунт, фраза
«УДАЛИТЬ ВСЁ»). Сайт: дашборд, рейсы/смены/расходы с фильтрами, финансы
(график по периоду + пресеты Неделя/Месяц/3/6/12), машины, водители,
Яндекс-карта (статусы: едет/стоит/двигатель работает; «стоит со вчера, 21:52 ·
уже 2 ч»), акт сверки Excel (форма 101 РС, выбор рейсов галочками), справочник
РЦ (импорт xlsx), реквизиты, админы, устройства (вечные сессии + удалённый
выход). GPS: парсер ЕГТС (координаты, скорость, курс, пробег-одометр трекера,
зажигание-эвристика), точки в БД, состояние машины, машины на карте, сравнение
пробега GPS↔одометр при вводе км, чужие терминалы не пишутся в БД, ночная
чистка (raw>7д, точки>60д). Известно: подзапись 17 в пакетах = точность GPS;
датчиков двери/ДУТ в потоке пока нет — появятся после установки, приёмник
залогирует (hex в логах).

## ЗАДАЧИ ЭТОЙ СЕССИИ (по приоритету, делать блоками)

### 1. Профиль машины в боте владельца (read-only, «первый этап»)
В боте владельца в «🚛 Мои машины» по нажатию на машину показать карточку:
гос.номер, марка, тип, норма л/100км, Stavtrack ID, сроки ОСАГО/ТО/тахографа,
GPS-статус сейчас (едет/стоит/двигатель работает, с какого времени — данные
уже есть в `vehicle_states` + `telemetry_service.motion_status_text`/
`smart_since_label`), пробег за сегодня по трекеру (`gps_mileage_for_period`),
кто сейчас в смене на ней. Редактирование в боте НЕ нужно (есть на сайте) —
только просмотр. Файл: `app/bots/owner_bot.py` (там уже есть список машин
`owner:vehicles` — добавить inline-кнопки по машине и хендлер карточки).

### 2. Уведомления бота — и админам тоже (у владельца 3 компьютера и 2 телефона)
Сейчас `notify_owner` шлёт только владельцу (owner.telegram_id). Второй телефон
— это ДРУГОЙ Telegram-аккаунт, добавленный как админ (таблица `admins`).
Сделать: уведомления владельцу дублировать всем админам кабинета (новая
функция в `app/bots/notifications.py`, например notify_owner → рассылка по
[owner.telegram_id] + admins.telegram_id, с try/except на каждого — админ мог
не нажать /start у бота, это не должно ронять рассылку; в этом случае молча
пропускать). ВАЖНО: /login-код шлётся только тому, кто его попросил — не
трогать. Опционально: флажок «получать уведомления» у админа (колонка
notifications_enabled, деф. true, миграция) — если просто, сделать.

### 3. Анти-миссклик водителя (не та машина / не то имя)
Контекст: водители обычно закреплены за машинами, но при старте смены выбирают
машину из списка — бывают миссклики. Сделать:
a) Миграция 0013: `drivers.default_vehicle_id` (FK vehicles, nullable) —
   «обычная машина». Редактирование: на сайте в карточке водителя (селект).
b) При старте смены в боте водителя: если выбранная машина ≠ default —
   водителю мягкое подтверждение («Обычно ты на Т772НХ178. Точно берёшь
   Е774ЕТ178?» — кнопки Да/Выбрать заново), владельцу — уведомление.
c) GPS-сверка «поехала не та машина»: джоба APScheduler каждые ~10 минут:
   для каждой АКТИВНОЙ смены смотреть `vehicle_states` ВСЕХ машин владельца.
   Если машина смены стоит (motion=stopped) > N минут, а ДРУГАЯ машина без
   активной смены при этом едет (motion=moving) — уведомить владельца:
   «⚠️ Возможно, перепутана машина: в смене Х (стоит), а едет Y (без смены)».
   Дедуп через `events` (не спамить чаще раза в 2 часа на пару машин).
   Это же закрывает будущий алерт «движение без смены» — реализовать как
   общий детектор в `scheduler_jobs.py` с использованием
   `telemetry_service.vehicle_control_signal` (уже есть).
d) Смена/рейс: возможность ИЗМЕНИТЬ водителя и машину на сайте задним числом —
   на страницах смены (`shift_detail.html`) и рейса (`trip_detail.html`)
   селекты «Водитель» и «Машина» + POST-роуты. При смене водителя у смены —
   пересчитать зарплату НЕ нужно автоматически (зарплата считается на лету из
   driver.salary_*), но проверить, что страницы её пересчитывают. Рейсы внутри
   смены при смене машины/водителя смены — обновить согласованно (спросить у
   владельца одним вопросом или обновлять каскадно с пометкой в events).

### 4. Дальше по плану (если останется ресурс / следующие сессии)
- Алерты владельцу: «едет после завершения рейса», «двигатель работает на

  активном рейсе».
- Геозоны РЦ (координаты уже в справочнике): авто-события «приехал/уехал с РЦ»,

- Утренняя сводка владельцу («Итоги вчера»: рейсы, деньги, аномалии).
- Датчики: владелец ставит датчики двери и топлива (ДУТ) на машины — когда
  появятся в ЕГТС-потоке (смотреть логи egts-receiver: unknown_sr / hex),
  добавить разбор подзаписей (дверь = дискретный вход / AD_SENSORS, топливо =
  LIQUID_LEVEL), вывести на сайт (уровень топлива на карточке машины и карте,
  алерт «дверь открыта вне РЦ», «резкое падение топлива на стоянке = слив»).

- Работай точечными правками, не переписывай работающее; после каждого блока —

  tests/test_migrations.py.
