# condur (TMS) — руководство для разработчика и тестировщика

Документ для человека, который видит проект впервые: что это, как устроено, как
запустить, **что и как тестировать** (особенно Excel), что стоит добавить и убрать.

---

## 1. Что это за продукт

**TMS для малых грузоперевозчиков** (ИП/ООО, 2–30 машин). Заменяет «Excel + блокнот».
Всё управление — через **два Telegram-бота** и **веб-кабинет**. Мобильного приложения и
SPA нет.

- **Бот владельца** — регистрация, справочники (водители, машины, маршруты), одобрение
  расходов, ввод пробега, код для входа в кабинет.
- **Бот водителя** — смены, рейсы, «выехал/сдал груз», гео, фото одометра, расходы, SOS.
  Почти все шаги за фича-флагами (по умолчанию выключены).
- **Веб-кабинет владельца** — дашборд, рейсы, водители, машины, финансы, карта,
  **акт сверки (Excel)**, реквизиты, админы.

Боль, которую решаем: воровство топлива, потерянные ТТН, несосчитанные рейсы, прибыль
«на глазок». Тариф — по числу машин (FREE/BASE/BUSINESS/PRO).

## 2. Стек и архитектура

| Слой | Технология |
|---|---|
| Язык | **Python 3.11** (см. `.python-version`) |
| Боты | **aiogram 3.13** (два бота в одном процессе) |
| Веб | **FastAPI** + **Jinja2** + **HTMX** + **Pico.css** (+ Chart.js с CDN) |
| БД | **PostgreSQL** через **SQLAlchemy 2.0 async** + **asyncpg** |
| Миграции | **Alembic** (сейчас до `0007`) |
| Планировщик | **APScheduler** (сводки, проверки документов, алерты) |
| Excel | **openpyxl** |
| Авторизация | **PyJWT** (JWT в httpOnly cookie) |
| FSM-хранилище | Redis опционально; сейчас **MemoryStorage** |
| Деплой | **Railway** |

**Один процесс** (`python -m app.main`) поднимает через `asyncio.gather`: два бота
(long polling) + FastAPI (uvicorn). Стек **заморожен** — без крайней нужды не менять.

## 3. Как запустить локально

```bash
# 1. Postgres (docker-compose.yml уже есть)
docker compose up -d db

# 2. Виртуальное окружение + зависимости
python3.11 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

# 3. Настройки
cp .env.example .env          # впиши токены двух ботов от @BotFather
#   JWT_SECRET сгенерируй: python -c "import secrets; print(secrets.token_urlsafe(48))"

# 4. Миграции
alembic upgrade head

# 5. Запуск (боты + веб)
python -m app.main            # веб на http://localhost:8000
```

**Переменные окружения** (`.env.example`):
`OWNER_BOT_TOKEN`, `DRIVER_BOT_TOKEN`, `DATABASE_URL`, `REDIS_URL` (пусто = MemoryStorage),
**`JWT_SECRET`** (на проде обязательно длинная случайная строка!), `PORT`,
10 × `FEATURE_*`, `RECEIPT_OCR_PROVIDER` + `ANTHROPIC_API_KEY`/`OPENAI_API_KEY` (OCR дормант),
`EGTS_PORT`/`EGTS_*` для отдельного Stavtrack-приёмника.

**Прод (Railway):** `startCommand = alembic upgrade head && python -m app.main` — миграции
применяются автоматически при деплое. Обязательно задать `JWT_SECRET` в Variables.

**Тесты:** `pytest` (нужен Python 3.11). Файлы — см. раздел 9.

## 5. Модель данных (основное)

- **owners** — владелец. Плюс **реквизиты Исполнителя для акта**: `executor_name`, `inn`,
  `ogrnip`, `legal_address`, `bank_name`, `bank_account` (р/с), `corr_account` (к/с),
  `bik`, `signer_name`. Таймзона, флаг уведомлений.
- **customers** — заказчики (для акта): наименование, ИНН, КПП, адрес, банк, р/с, к/с,
  БИК, № договора + дата, подписант.
- **admins** — доп. администраторы кабинета (полный доступ): `owner_id`, `telegram_id`, `name`.
- **drivers** — водители: ФИО, телефон, тип ЗП (`per_km/per_trip/percent/fixed_per_shift/fixed_per_month`),
  ставка, суточные, ожидаемое время старта смены, invite_token.
- **vehicles** — машины: гос.номер, марка, тип, норма л/100км, сроки ОСАГО/ТО/тахографа,
  `stavtrack_object_id` для привязки ID объекта из Stavtrack.
- **shifts** — смены: статус, старт/конец, одометр старт/конец (+ фото), `distance_km`
  (вычисляемая = конец − начало), `is_manual`.
- **trips** — рейсы: статус, откуда/куда, груз, № ТТН + фото, выручка, топливо, прочие
  расходы, `profit_rub` (вычисляемая = выручка − топливо − прочие), `is_manual`.
- **trip_documents** — документы к рейсу (байты в Postgres).
- **expenses** — расходы водителя: категория (`fuel/repair/parking/fine/toll/other`),
  сумма, фото чека (Telegram file_id и/или байты), статус (`pending/approved/rejected`).
- **events** — лог всех действий (JSONB payload).
- **route_templates** — шаблоны маршрутов.
- **distribution_centers** — справочник РЦ: название, адрес, алиасы, координаты; акт сверки
  использует адрес при совпадении с `Trip.destination`.
- **vehicle_telemetry_raw_packets** — сырые EGTS/TCP-пакеты Stavtrack; первый слой интеграции.
- **vehicle_telemetry_points** — будущие разобранные GPS-точки.
- **vehicle_states** — будущее последнее состояние машины для бота/кабинета.
- **manual_entries** — ручные доходы/расходы
  владельца (страница Финансы). **subscriptions** — тариф. **daily_summaries** — кэш сводок.

Всё скоупится по `owner_id`. Пробег есть только на уровне **смены** (не рейса — одометр
снимается по фото на старте/конце смены, а в смене может быть несколько рейсов).

## 6. Авторизация (важно для тестирования)

1. Владелец в боте: `/start` → онбординг (компания, телефон) → создаётся `Owner`.
2. `/login` в боте → 6-значный код (в памяти, живёт 5 минут).
3. Веб `/login` → вводит **Telegram ID + код** → JWT в httpOnly cookie (7 дней).
4. **Админ:** владелец добавляет его **по Telegram ID** (Реквизиты → Администраторы). Админ
   открывает бот владельца → `/login` → код → входит на веб `/login` → получает **полный
   доступ** к кабинету владельца.
5. JWT хранит `owner_id` + `tid` (кто вошёл). `current_owner` при каждом запросе проверяет,
   что админ ещё не удалён → **удаление админа отзывает доступ мгновенно**.

⚠️ Коды хранятся в памяти процесса — при рестарте теряются (это ок, TTL 5 мин). Уже выданные
JWT переживают рестарт. `JWT_SECRET` на проде должен быть длинным и секретным.

## 7. Функции по модулям

### Бот владельца
Онбординг; водители (добавить/список/статус); машины (добавить/список, лимит по тарифу);
шаблоны маршрутов; **ввод пробега** по фото одометра (число вписывает владелец, кнопка
«📍 Указать пробег», валидация ≤7 цифр / конец≥начала / ≤5000 км, одноразово); одобрение
расходов; `/login`, `/timezone`, `/tariffs`, `/calc`, `/cancel`, `/help`.

### Бот водителя (за фича-флагами)
Начать/Конец смены, Новый рейс, Выехал, Сдал груз (+ гео), SOS, Документ (ТТН), Расход,
«➕ Добавить смену/рейс» задним числом. Выручку указывает после сдачи («правило первого»).
Флаги (`app/config.py`, по умолчанию **выключены**, кроме `CARGO_GEOLOCATION`, `RECEIPT_OCR`;
`ODOMETER_PHOTO` в примере off): скрывают кнопки/шаги без дублирования кода.

### Веб-кабинет (роуты в `app/web/router.py`)
- `/` дашборд (KPI, «Требуют внимания», графики выручка/прибыль, пончик расходов, машины
  в работе, превью карты) + `/api/dashboard-chart` (периоды 7д/30д/**6м**/12м).
- `/trips` рейсы (фильтры, пагинация, добавить вручную) — HTMX.
- `/drivers` водители — **карточки**, показатели **за всё время**, инлайн-правка (HTMX).
- `/vehicles` машины — **карточки** (P&L за месяц, статус «в рейсе»), `/vehicles/{id}/stats`.
- `/finances` доходы/расходы, ручные записи, **график «Денежный поток · 6 мес»**,
  **«Прибыльность направлений»**, `/finances/export.xlsx`.
- `/expenses` все расходы (фильтр, пончик по категориям), `/expenses/{id}` правка (сумма/
  категория/статус/фото — **работает даже после одобрения**).
- **`/acts` акт сверки** → `/acts/export.xlsx` (см. раздел 8).
- `/requisites` реквизиты Исполнителя + заказчики + **администраторы**.
- `/routes` направления + **справочник РЦ** (ручное добавление/правка, импорт `.xlsx`,
  мягкое скрытие), `/map` карта водителей (Leaflet + OSM, без ключей).
- `/login`, `/logout`.

Дизайн-система `.sp-*` (карточки/KPI/графики) встроена в `<head>` страниц через партиал
`_sp_styles.html` (чтобы не зависеть от кэша `cabinet.css`).

## 8. 🟢 EXCEL — ПОДРОБНО (главное для проверки)

В проекте **два Excel-экспорта**:

### 8.1 Акт сверки — `app/services/act_service.py`
Это ключевой документ. **Один акт = один лист** со всеми выбранными рейсами за период.

**Поток:**
```
/acts (страница)                         /acts/export.xlsx (GET)
  период + заказчик + название +           собирает: реквизиты Исполнителя (owner),
  № акта + дата + ЧЕК-ЛИСТ рейсов   ──►     реквизиты Заказчика (customer), выбранные
  (галочки: какие включить)                рейсы (trip_ids) → build_act_101rs() → .xlsx
```

**Главная функция:** `build_act_101rs(title, act_number, act_date, period_from,
period_to, executor: dict, customer: dict, rows: list[dict]) -> Workbook`.

**Что на листе (сверху вниз):**
1. Заголовок: **`{title} № {act_number} от {дата} г.`** (title по умолчанию «Акт», на
   сайте поле «Название документа» = «Акт сверки»).
2. **Исполнитель:** ФИО/ИП, ИНН, ОГРНИП, адрес, банк, р/с, к/с, БИК (из `owner`).
3. **Заказчик:** наименование, ИНН, КПП, адрес, р/с, банк, БИК, к/с (из `customer`).
4. **Основание:** Договор № … от … (из `customer`).
5. Таблица: **№ · Наименование работ, услуг · Кол-во · Ед. · Цена, руб · Сумма, руб**.
   Строка = рейс, описание: `{дата}. {откуда}- {куда}, {гос.номер}, {водитель}.`
6. **Итого**, затем «**Всего оказано услуг N. На сумму X руб. {сумма прописью}**».
7. «**Без налога (НДС)**», закрывающий текст, **подписи сторон**.

**Формат — «точь-в-точь как образец 101 РС»** (константы `_ACT_*` в act_service):
шапка таблицы **без заливки** (чёрный жирный текст), деньги **`#,##0.00`** (без ₽),
шрифт **9 пт**, **чёрные тонкие рамки**. Сумма прописью — `rubles_in_words()` (свой
русский конвертер, обрабатывает рубли/копейки, род «одна тысяча» и т.п.).

**Выбор рейсов:** `trip_ids` в query. **Пусто = все рейсы периода** (это поведение прямой
ссылки без чек-листа). Отмеченные галочки → только они.

**Источник данных строк:** `Trip.origin` / `Trip.destination` — свободный текст, что ввёл
водитель/владелец. Если на `/routes` заполнен справочник РЦ, `Trip.destination`
нормализуется по названию/адресу/алиасу и в акт подставляется канонический адрес РЦ.
Если совпадения нет — остаётся свободный текст.

**Мёртвый код:** старый `build_acts_workbook` (листы по РЦ) больше не используется —
**можно удалить** вместе с `_safe_sheet_title`.

### 8.2 Экспорт финансов — `/finances/export.xlsx`
Проще: суммарные доходы/расходы + листы записей и рейсов за период. Код прямо в
`router.py` (`_build_finance_workbook`, `_fill_entries_sheet`, `_fill_trips_sheet`).

### 8.3 Что тестировщику проверить по Excel (ПРИОРИТЕТ)

**Акт сверки:**
- [ ] Реквизиты Исполнителя и Заказчика попадают в шапку **правильно** (ИНН/КПП/банк/адрес).
- [ ] Если реквизиты **не заполнены** — акт всё равно генерится, но с пропусками (страница
      показывает предупреждение). Проверить оба случая.
- [ ] Если **нет заказчика** — блок Заказчика пустой, акт генерится.
- [ ] Колонки строго как в образце: № · Наименование · Кол-во · Ед. · Цена · Сумма.
- [ ] **Итого = сумме выбранных** рейсов. Снял галочку → сумма и список меняются.
- [ ] **Сумма прописью** верна на разных суммах: с копейками (1234,56), ноль, 1 рубль,
      21 (двадцать один рубль), 178 000, 292 000. Род и склонения.
- [ ] **Название документа** из поля отражается в заголовке («Акт сверки», «Акт», своё).
- [ ] В акт попадают **только отмеченные** рейсы. **Особый кейс:** снять ВСЕ галочки →
      кнопка скачивания блокируется. Прямая ссылка без чек-листа по-прежнему выгружает все
      рейсы периода.
- [ ] Живой счётчик «Выбрано: N на сумму X ₽» на странице совпадает с итогом в файле.
- [ ] Файл **открывается без ошибок** в Excel, LibreOffice и Google Sheets.
- [ ] Имя файла с кириллицей («Акт_…») скачивается корректно (заголовок RFC 5987).
- [ ] Формат: шапка без синей заливки, деньги «19 000,00» (2 знака, без ₽), мелкий шрифт,
      чёрные рамки — визуально как присланный образец `101 РС`.
- [ ] Длинные адреса/описания не «рвут» строку (перенос по ширине колонки).
- [ ] Период без завершённых рейсов с выручкой → страница пишет «нет рейсов», акт пустой.

**Экспорт финансов:**
- [ ] Суммы совпадают с тем, что на странице Финансы за тот же период.
- [ ] Открывается в Excel/LibreOffice.

- **Тестовый мусор в боевой БД:** рейсы/машины/водители вроде «mmm», «ттт», «оыраовраор»,
  «оооо → оооо» — из-за них акты и статистика выглядят несерьёзно. Почистить.
- `.venv-tools/` — временное окружение для локальных проверок, уже в `.gitignore`.

## 13. Известные проблемы / подводные камни

- **Локально без Postgres + токенов не поднять** живьём. Быстрая проверка — `pytest` и
  компиляция; шаблоны рендерятся в тестах без БД.
- Пробег есть **только по смене**, не по рейсу (следствие модели «фото-одометр за смену»).
- Карточки **машин** — показатели **за месяц** (помечено), карточки **водителей** — **за
  всё время**. Не перепутать при проверке (0 у машины в новом месяце — это норма).
- Если справочник РЦ пустой или нет совпадения по алиасу, акт использует свободный текст
  `Trip.destination`.
- `JWT_SECRET` по умолчанию `change-me-in-production` — **на проде обязательно сменить**.
- Коды входа и FSM — в памяти процесса (MemoryStorage): при рестарте теряются.
