# План: «GPS-достоверность рейса» (сверка слов водителя с GPS)

Дата: 2026-07-08. Автор плана — для следующей сессии/разработчика. Кода пока нет,
это проектирование. Реализовывать МАЛЫМИ блоками, каждый с тестами (см. фазы).

## 1. Идея простыми словами

Водитель НАЖИМАЕТ кнопки: «Начал смену», «Новый рейс», «Выехал», «Выгрузка»,
«Сдал груз». GPS (Stavtrack) НЕЗАВИСИМО показывает, что машина делала на самом
деле: стояла у склада, поехала, приехала на РЦ, стояла под выгрузкой, уехала.

Система должна по каждому рейсу сравнить «что сказал водитель» с «что показал
GPS» и вынести вердикт: **совпадает / есть расхождения / нельзя проверить**, с
разбивкой по шагам и временем. Это делает данные доверенными без ручной сверки.

## 2. Что уже есть (фундамент — не переделывать)

- **Отметки водителя** (таблица `events`, у каждой время `created_at`):
  `shift_started`, `trip_created`, `trip_in_transit` («выехал»),
  `trip_unloading`, `trip_completed`, `waybill_uploaded`.
- **GPS-реальность**: геозоны РЦ → `rc_arrived` / `rc_departed` (со временем
  фактической остановки `parked_since` и отъезда); `vehicle_telemetry_points`
  (координаты/скорость по времени); `vehicle_states` (текущее состояние).
- **Уже сверяется**: `trip_rc_confirmed` / `trip_rc_mismatch` (тот ли РЦ);
  `moving_without_shift_alert` (поехал без смены); детектор «перепутана машина».

Итого 80% сигналов уже пишется — нужен слой, который их СОПОСТАВЛЯЕТ по одному
рейсу и выдаёт понятный вердикт.

## 3. Что сверяем (шаги рейса → проверка по GPS)

Для каждого рейса берём цепочку отметок водителя и проверяем каждую окном ±N мин:

| Отметка водителя | Что должен показать GPS | Вердикт шага |
|---|---|---|
| Начал смену (T0) | машина у склада / вот-вот тронулась | ✅/⚠️/❔ |
| Выехал (T1) | машина реально тронулась со склада ≈ T1 | ✅/⚠️/❔ |
| (в пути) | был `rc_arrived` на РЦ назначения | ✅ совпал РЦ / ⚠️ другой РЦ / ❔ |
| Выгрузка (T2) | машина стоит в геозоне РЦ ≈ T2 | ✅/⚠️/❔ |
| Сдал груз (T3) | машина на РЦ / вскоре `rc_departed` ≈ T3 | ✅/⚠️/❔ |

Вердикт шага:
- **✅ совпало** — GPS подтверждает событие в пределах окна.
- **⚠️ расхождение** — GPS показывает другое (не там / не в то время / порядок нарушен).
- **❔ нет данных** — GPS молчал в этом окне (проверить нельзя — НЕ вина водителя).

**Итог по рейсу:**
- **✅ Подтверждён GPS** — все проверяемые шаги ✅.
- **⚠️ Есть расхождения** — хотя бы один ⚠️ (показать какие).
- **❔ Мало GPS-данных** — большинство ❔ (например, не было связи).

## 4. Честная обработка проблем (главное — не наказывать зря)

Владелец прав: у водителя может не быть интернета на складе, GPS может лагать.
Правила, чтобы система была СПРАВЕДЛИВОЙ:

1. **Нет GPS в окне → «нет данных», НЕ «расхождение».** Молчание трекера или
   потеря сигнала никогда не трактуется как обман.
2. **Отметка пришла с задержкой (не было интернета) — это норма.** Проверяем не
   «время нажатия = время GPS», а что **физическое событие вообще произошло** и
   что **порядок отметок совпал с порядком GPS-событий**. Если водитель нажал
   «выгрузка» на час позже, но GPS подтверждает, что машина реально была на РЦ —
   это ✅ с пометкой «отмечено с задержкой», а не ⚠️.
3. **Большие окна допуска** (настраиваемые, старт ~30–60 мин) — город, пробки,
   очередь на КПП, поздние нажатия.
4. **«Пачка задним числом»**: если несколько отметок пришли почти одновременно
   (водитель настучал всё разом, когда появилась связь) — помечаем «внесено
   пачкой» как ИНФО, не как нарушение. Считаем по GPS, а не по времени нажатий.
5. **Только COMPLETED-рейсы** оцениваем полностью (у активного данные ещё копятся).

## 5. Техническая реализация (архитектура)

**Ядро — чистая функция без БД (тестируемая):**
`app/services/trip_verification.py`
```
def verify_trip(marks: list[Mark], gps_events: list[GpsEvent],
                windows: Windows) -> TripVerdict
```
где `Mark` = (тип, время) отметки водителя, `GpsEvent` = (тип, время, rc_id)
из геозон/точек. Возвращает список шагов с вердиктами + общий вердикт + флаги
(«задержка», «пачка», «нет данных»). Вся логика окон/сравнения — здесь, чтобы
покрыть юнит-тестами (как `test_rc_geofence.py`), БЕЗ поднятия сайта/БД.

**Обвязка (тонкая):**
- `router.trip_detail` собирает marks (события рейса) и gps_events (rc_arrived/
  rc_departed по машине в окне рейса — уже собираются для таймлайна) → зовёт
  `verify_trip` → отдаёт вердикт в шаблон. **Без миграции.**
- Показ в карточке рейса: бейдж «GPS-достоверность: ✅/⚠️/❔» + раскрытие по шагам
  (каждая отметка водителя и её GPS-проверка со временем и разницей).

## 6. UI

- **Карточка рейса**: бейдж вердикта вверху + блок «Сверка с GPS» (таблица шагов:
  отметка · время водителя · факт по GPS · разница · вердикт).
- **Список рейсов**: маленькая иконка ✅/⚠️/❔ в строке.
- **Статистика → по водителям**: колонка «Расхождений с GPS» (сколько рейсов с ⚠️)
  — это и есть «рейтинг честности» водителя, без обид: показывает факты.

## 7. Фазы (каждая — отдельный безопасный блок)

- **Фаза 1** — `trip_verification.py` (чистое ядро) + юнит-тесты на все случаи:
  совпало / не тот РЦ / не то время / задержка отметки / пачка / нет GPS. НИКАКОГО
  UI и БД. Это «мозг», его можно вылизать тестами.
- **Фаза 2** — бейдж и блок «Сверка с GPS» в карточке рейса (`trip_detail`, читает
  уже собираемые события). Без миграции.
- **Фаза 3** — иконка в списке рейсов + «расхождений с GPS» в статистике по водителям.
- **Фаза 4 (опц.)** — снимок вердикта в момент завершения рейса (событие
  `trip_gps_verdict` или колонка) — чтобы список не пересчитывал каждый раз.
- **Фаза 5** — калибровка окон допуска на реальных данных (недельки две наблюдать).

## 8. Что НЕ делаем

- Не блокируем и не штрафуем водителя автоматически — только показываем факты
  владельцу. Решение — за человеком.
- Не считаем «нет GPS» обманом.
- Не тянем платные сервисы/тяжёлые зависимости — всё на текущих `events` + GPS.

## 9. Оценка объёма

по мере надобности. Риск низкий: ядро изолировано и тестируемо, обвязка читает
уже существующие события, миграции не обязательны до Фазы 4.
