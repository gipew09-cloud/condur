# Condur — TMS для малого автопарка

Система управления небольшим автопарком рефрижераторных грузоперевозок (свежие
продукты со складов на распределительные центры). Владелец в реальном времени видит
где машины и что делают водители, сколько заработано/потрачено, и получает уведомления
о важных событиях. От водителя требуется минимум — пара кнопок в Telegram.

Три поверхности: **бот владельца** (Telegram), **бот водителя** (Telegram),
**веб-кабинет** (FastAPI).

## Стек

- **Python 3.12**, полностью async.
- **aiogram 3** — оба Telegram-бота (long polling).
- **FastAPI + Jinja2** — веб-кабинет (server-side рендеринг, без SPA).
- **SQLAlchemy 2 (async) + PostgreSQL** — данные. **Alembic** — миграции.
- **Redis** — FSM-хранилище диалогов ботов.
- **APScheduler** — фоновые задачи (сводки, детекторы, геозоны РЦ, чистка телеметрии).
- **Leaflet / Яндекс.Карты** — карта машин.
- Тесты: **pytest + aiosqlite** (in-memory SQLite).

## Структура

```
app/
  main.py            вход основного сервиса (боты + веб + планировщик)
  run.py             выбор роли по SERVICE_ROLE (main | egts)
  models.py          модели БД (SQLAlchemy)
  config.py          настройки из окружения
  bots/              owner_bot, driver_bot, клавиатуры, тексты, уведомления, middlewares
  services/          бизнес-логика: смены, рейсы, расходы, зарплата, телеметрия,
                     геозоны, планировщик, финансы, время
  telemetry/         приёмник GPS: egts_receiver (TCP), парсеры egts и wialon
  web/               router (FastAPI), templates (Jinja2), static
tests/               pytest-набор
alembic/             миграции
```

## Запуск локально

```bash
# 1. Postgres — docker-compose.yml уже есть
docker compose up -d

# 2. Окружение (Python 3.12; для тестов удобен uv)
uv venv --python 3.12 .venv
uv pip install --python .venv/bin/python -r requirements.txt pytest pytest-asyncio aiosqlite

# 3. Настройки — скопировать .env.example → .env, заполнить токены/БД.
#    JWT_SECRET: python -c "import secrets; print(secrets.token_urlsafe(48))"

# 4. Миграции
alembic upgrade head

# 5. Запуск (боты + веб + планировщик)
python -m app.run
```

Тесты:

```bash
.venv/bin/python -m pytest tests/
```

## Деплой (Railway)

Авто-деплой из GitHub. Сервисы:

- **condur** — основной (боты + веб + планировщик). Start: `python -m app.run`.
- **egts-receiver** — только TCP-приёмник GPS. Тот же образ, но `SERVICE_ROLE=egts`
  и проброшенный TCP-порт (`EGTS_PORT`, по умолчанию 9000).
- **Postgres**, **Redis**.

`SERVICE_ROLE`: пусто/`main` — основной сервис; `egts` — только приёмник.

## Телематика (GPS)

Данные машин идут через ретрансляцию **Stavtrack** на наш TCP-приёмник
(`app/telemetry/egts_receiver.py`), который на одном порту понимает два протокола:

- **EGTS** (бинарный) — координаты, скорость, пробег прибора.
- **Wialon IPS** (текстовый) — то же плюс датчики: напряжение бортсети, зажигание,
  пробег (`totalDistance`, метры), моточасы.

Точки пишутся в `vehicle_telemetry_points` (история), последнее состояние — в
`vehicle_states` (быстрый слой для карты). Машина сопоставляется по
`vehicles.stavtrack_object_id` (задаётся в кабинете на `/vehicles`).

Подключение новой машины в Stavtrack: раздел ретрансляций → протокол `egts` или
`wialon`, TCP, адрес и порт сервиса `egts-receiver`, выбрать нужный объект; в кабинете
заполнить у машины `Stavtrack ID`.

**Важно (определение зажигания):** «двигатель работает» определяется по напряжению
бортсети (генератор заряжает → ~28 В у 24-вольтовой сети), а не по сырому биту `ign` —
он у части трекеров залипает. Так же считает и сам Stavtrack. Подробнее — в комментариях
`app/services/telemetry_service.py`.

## Принципы

- Данные не удаляются: события/деньги — вечно, GPS-точки — ограниченный срок.
- Честность цифр: нет данных → пишем «нет данных», а не 0; деньги за простой не
  списываем автоматически.
- Всё в UTC, показываем в часовом поясе владельца.
- На каждый баг — регресс-тест; проверяем, что тест ловит баг, откатом фикса.

Текущий план развития и известные ограничения — в [ROADMAP.md](ROADMAP.md).
